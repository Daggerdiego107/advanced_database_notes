## Topics covered
- PL/SQL triggers (`BEFORE INSERT`, `BEFORE UPDATE`, `BEFORE DELETE`)
- Row-level trigger execution (`FOR EACH ROW`)

- Using `:NEW` and `:OLD` pseudorecords for data manipulation and validation
- Exception handling in triggers using `RAISE_APPLICATION_ERROR` and `WHEN OTHERS`

## What I understood
- Triggers can be used not just for auditing, but for enforcing strict access control at the database level. For example, restricting `UPDATE` and `DELETE` operations based on the current session `USER`.
- The `:NEW` pseudorecord allows for automatic population of auditing columns (like `UPDATE_DATE` and `UPDATED_BY_USER`) right before a record is committed, preventing users from spoofing metadata.
- The `:OLD` pseudorecord is crucial for validation, allowing the database to compare the current session user against the user who originally created the record before permitting an update.
- General database errors within a trigger can be gracefully trapped with an `EXCEPTION WHEN OTHERS` block, and then bubbled up to the application using `RAISE_APPLICATION_ERROR`.
- This approach to row-level auditing and security is highly relevant for refining the Oracle database automated triggers in OctoTask, ensuring that backend data integrity remains intact regardless of the frontend client.
